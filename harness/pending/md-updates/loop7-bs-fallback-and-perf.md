## 1. BS 섹션 배정 폴백 카나리 (검증 2 부수 관측)
`dart._assign_sections` 는 IFRS 앵커 금액 포함관계로 자산/부채/자본을 닫는다. 구조가 깨져
못 닫으면 '직전 배정 섹션'으로 폴백하고 `fallback_used=True` 로 정직 표기한다(지어내지 않음 —
원칙 3·4 준수). 검증방 실측(캐시 4사: 00111704·00126380·00153861·00164742) 전부
**fallback_used=False**(깔끔히 닫힘, 오배정 없음).

- 위험 표면: 폴백이 발동하면 말단 계정이 '틀린 섹션'(예: 자본 계정이 부채 열)으로 갈 수 있다.
  화면 T배치가 잘못되어도 fallback_used 로 알려지지만, 사용자가 그 플래그를 못 볼 수 있다.
- 처방: 라이브에서 fallback_used=True 가 관측되는 회사를 만나면, 검증방이 그 회사 BS 를
  케이스로 tests/invariants 에 고정하고 섹션 배정을 점검한다(현재는 트리거 사례 없음 —
  발동 시 발효). 또는 화면에 폴백 시 '구조 추정(확인 요망)' 배지를 노출(원칙 7 투명성).

## 2. 성능 갭 (판단 왜곡 아님 — 기록)
개발방이 정직하게 보고: 병렬화 후에도 태영 웹 실행 **~270초(4.5분)**. 원인은 Opus adaptive
thinking 의 청크당 지연(~1~2분)이 상한이고 병렬(동시6)이 겹쳐 ~3~4배 단축했을 뿐, '1분 내'
목표엔 미달. 이는 병렬 불변(결과 동일, 검증 통과)과 무관한 '속도'의 문제다.

- 순차 대비 병렬 우위는 검증방이 실측 확인(병렬==순차, 유실 0). 속도만 미달.
- 처방 후보(스코프 밖, 다음 성능 루프): thinking 완화/청크 크기·동시성 상한 조정은 판단
  품질과 트레이드오프이므로, 바꾸면 검증방이 판단 불변식(direction/paraphrase/magnitude 등)을
  다시 태워 품질 회귀가 없는지 확인해야 한다. 지금은 품질 우선으로 속도를 양보한 상태.

## 개발 반영 (루프11 — 개발방): #1(BS 폴백 카나리) 트리거 케이스 발생·수정·고정. #2(성능)는 잔류.
라이브에서 fallback_used=True 가 CJ제일제당에서 관측됐다(이 처방이 예고한 '발동 시 발효' 상황).
원인 규명(추측 아니라 캐시 실측): CJ BS 가 비정형 순서(자본이 부채보다 먼저 + grand 총계 중간) +
매각예정 처분집단 부채(중간 leaf)로 부채총계가 산술 미탐지 → running 오염 → 부채 블록 못 닫혀
폴백이 부채를 '자본'에 밀어넣음(부채 0·자본 오염). 수정: DART/IFRS 표준 구간합계 계정ID 마커로
구조총계를 보완(이름표 아님 — 원칙 1)해 누적 오염 제거 + 폴백을 '인접 밀어넣기'에서 '구분
미확정(None)'으로 바꿔 정직 노출(원칙 4·7). 실측: CJ 부채 0→18, 미확정 0·폴백 False, 나머지 4사
회귀 0(is_total==is_sub). 화면도 폴백/미확정 시 '확인 요망' 배지. 이 처방의 '폴백 시 배지 노출'
제안도 반영됨. tests/unit/test_bs_section_disposal_group.py 로 CJ 를 회귀 케이스로 고정.
- **#1 은 개발 시공 완료.** 단 이 파일은 #2(성능 갭)가 남아 통째로 비우지 않는다 — 성능은 판단
  품질 트레이드오프라 별도 성능 루프·설계자 몫으로 잔류.

**검증방 몫(원칙 10):** 이 처방의 본래 취지대로, CJ BS 를 tests/invariants/ 에 섹션 배정 불변식
