- **표시강등 메커니즘은 CLEAN.** 강등의 유일한 트리거는 `subject_role=='타사언급'`(값 비교)이고,

  제목·키워드로 '계산'하지 않는다(오직 parse_batch 가 LLM 값을, load_full 이 덤프 값을 실어 나름).
  → 원칙 1(하드코딩) 위반 없음.

  C 도 삭제·접힘이 아니라 표시분(shown) 하단에 남아 접근 가능. 정합(수집=표시+접힘)·증발 0 유지.

- 검증방 소유 불변식 `tests/invariants/test_subject_role_demotion.py`(14건) 시공, red→green

## 검증 못 한 것 (정직 — 원칙 4)
표시층은 'LLM 이 role 값을 주면 원칙대로 반영하는가'만 검증됐다. **'LLM 이 role 을 제대로
매기는가'(분류 정확도)는 라이브 API 가 필요**한데, 루프10 프롬프트 변경으로 판단 캐시가
바뀌었고 크레딧 상황상 실측을 못 했다. 이게 이 기능의 핵심 위험(중요한 걸 C로 내림)이 실제로
발생하는 유일한 경로다 — 코드가 아니라 LLM 오분류에서만 난다.

## 처방
1. **(재현성 — 다음 검증방이 $0 로 분류 정확도 재검) subject_role 을 실은 예시 덤프 커밋.**
   total-verify-observations #1 과 같은 방향: 악재·비교언급이 풍부한 대표 회사의 full_dump 를
   out/results 에 커밋해 두면, 다음 검증방이 라이브 없이 'LLM 이 실제로 무엇을 A/B/C 로 달았는지'를
   읽어 정확도(오분류로 인한 놓침)를 재검할 수 있다.
2. **(라이브 체크리스트 — 크레딧 확보 시 실측) 정확도 실측 케이스(기대 라벨):**
   - "건설사 5년 부채비율…이 회사 489%"(복수사 나열·실질 수치) → **B**(C면 놓침 실패).
   - 타사 실적 브리핑에 이 회사 과징금 한 줄 → **C**.
   - 지분46% 자회사 관리종목 지정(제목에 모회사명 없음) → **A/B**(C면 놓침 실패).
   - 애매 경계 → **B/미분류**(C 아님, 놓침 방지).
3. **(신뢰경계 canary — 검증방 소유 불변식 후보) subject_role 비정규 값 감지.**
   `is_peripheral_mention` 은 정확히 "타사언급"만 True. 구조화 출력 enum 이 라이브 경로에선 이를
   강제하나, 프롬프트/enum 이 드리프트하면 비정규 값이 **조용히 '미분류'로 no-op**(원칙 7 쪽으로
   안전하나 '기능이 조용히 꺼짐'). 비어있지 않은 비정규 subject_role 을 감지/경고하는 canary 를
   둘지(설계자 판단; 검증방이 불변식으로 못박을 수 있음).
4. **(#5 연계) C 표기가 생겼으니 pending total-verify-observations #5('지엽 언급→자본총계 형식적
   계정연결')를 'C 는 계정 기권 쪽으로' 정책으로 다룰지.** 설계자/검증방 몫 — 개발방 자의 금지(원칙 10).

_비개발자 요약:_ 루프10에서 '남의 기사에 스친 언급'을 화면 아래로 내리는 기능이 안전하게(중요한
기사를 잘못 내리지 않고, 제목·키워드 매칭도 쓰지 않고, 무엇도 숨기지 않고) 작동함을 독립 검증으로
확인했습니다. 다만 'AI 가 실제로 각 기사를 제대로 A/B/C 로 분류하는가'는 유료 API 가 있어야
확인할 수 있어 이번엔 못 했습니다 — 그래서 (1) 대표 회사 결과를 파일로 저장해 두어 다음에 돈 없이
재검하게 하고, (2) 크레딧이 생기면 확인할 케이스 목록을 남기고, (3) AI 가 이상한 분류값을 주면
조용히 기능이 꺼지지 않도록 경고를 달지, 를 다음 과제로 올립니다.
