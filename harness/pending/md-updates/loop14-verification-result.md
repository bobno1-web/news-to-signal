- 발의: 검증방 / 루프14 독립 검증(개발방 보고 불신·직접 재현) / 2026-07-21
- 방법: 결정론(무API·무과금) + 읽기전용 라이브 DART 프로브(무비용·비파괴). 라이브 LLM(Haiku/Opus)은
  실비용이라 사용자 고지 전까지 미실행 — 해당 항목은 아래에서 '라이브 필요'로 명확히 분리.
- 요약 판정: **B1(국면 삼킴)·B2(분모 기수) 두 blocker 모두 해소 방향이 실물로 확인됨.** 비용 최적화는
  판단을 흔들지 않음(구조적). 단 잔여 결함 3건(하나는 검증방 불변식 미완, 둘은 코드 갭)을 아래 처방.

## 검증 1 — 국면 보존 (B1) : 코드 메커니즘 PASS / 라이브 거동 미검

- **하드코딩 없음(원칙 1 통과):** prescreen 군집 코드는 Haiku 가 준 정수 event_id 로 '묶기만' 한다
  (prescreen.py:254-285) — 단계·국면 문자열 매칭 없음. `_SYSTEM`·schema·system.txt 의 '심의/재판/확정'
  류는 전부 **자연어 예시**이며 "고정 단계 목록으로 나누지 말라(원칙 1)" 명시(prescreen.py:82-84).
  grep 결과 코드층 단계표(_STAGES/_PHASES) 0건. 반기·분기 코드(11012/13/14) 0건.
- **국면 보존 메커니즘 결정론 실측(monkeypatch _screen, 무API):** 담합 5국면에 서로 다른 event_id 를
  주면 → **5국면 전부 각각 대표(survivor)로 생존**(과징금500억확정 포함 — 이전엔 '중복'으로 삼켜졌던
  국면). 같은 국면 매체중복(같은 event_id) 1건만 접힘(비용 억제 유지). 부산/울산 화재 미병합. 정합
  (수집=대표+접힘) 성립. distinct event_id 수 == survivors 수.
- **역할 분리 확인:** 입구(prescreen event_id)=같은 국면 중복만 접음 / 표시층(schema.event_group)=같은
  사안의 국면들을 '각각 남기되 묶어 보임'. 두 층이 정반대로 싸우던 루프11/13 충돌이 §A(프롬프트 반전)로
  정합됨. 처방 prescreen-must-not-merge-event-phases §A = 반영 확인.
- **신규 불변식 세움(검증방 소유):** `tests/invariants/test_prescreen_phase_preservation.py`(4건, 결정론
  monkeypatch — 무API). 위 코드 메커니즘을 영구 회귀망으로 고정. 키 차단 상태에서 4/4 통과.
- ★**라이브 필요(미검, 실비용):** '실제 Haiku 가 5국면에 서로 다른 event_id 를 매기는가'(프롬프트 거동)는
  유료 Haiku 호출이라 이 검증에서 미실행. 개발방 changelog 는 라이브로 '과징금확정 통과(True)'를 봤다고
  보고하나, 검증방이 재현하지 못했다(억지 통과 주장 안 함). → 처방 A.

## 검증 2 — 분모 기수 (B2) : PASS (읽기전용 라이브로 확증)

- `_YEARS` 하드코딩 제거 → `_years()`가 `datetime.now().year`부터 역순 동적 계산. 라이브 실측:
  오늘(2026-07-21) `_years()`=['2026','2025','2024','2023','2022']. 특정 회사·연도 하드코딩 0(원칙 1).
- **연간 사업보고서만:** `_REPRT_ANNUAL="11011"`, `_fetch_latest`가 이 코드만 사용. 반기·분기 미혼입.
- **폴백 정상:** 읽기전용 프로브에서 FY2026 → status 013(미공시) → 직전 연도로 하강 확인.
- **★제19기 수신 확증(읽기전용·캐시 미변조):** `_fetch_latest("00635134",["2025"])` 직접 호출 →
  **FY2025 = 제19기, rcept_no 20260316001116, 자산총계 29,744,790,534,000(29.74조)** 수신. 이는 감사인이
  발단에서 제시한 "제19기 29.7조"와 **정확 일치** — B2 가 staleness 를 실제로 해소함을 실물로 입증.
  (현 캐시는 FY2024 그대로 둠 — 픽스처 보존 위해 `_statement_rows` 대신 read-only `_fetch_latest` 사용.)
- **캐시 무효화 로직:** 코드 정독 + 실입력 확인으로 검증. 캐시 bsns_year=2024 < latest 2026 → stale →
  newer=['2026','2025'] 프로브 → FY2025 rcept(2026...) ≠ 캐시 rcept(2025...) → 갱신. 각 단계 입력이 실물로
  참임을 확인(rcept 상이·FY2025 존재). 단 end-to-end 캐시 쓰기는 픽스처 변조 방지로 미실행(라이브 필요).
- **세 분모 동일 보고서:** `_extract_financials`가 한 rows 에서 매출·자산총계·자기자본을 전부 뽑음(구조
  보장). FY2025 프로브에서 셋 다 제19기 값(29.74/27.34/11.55조)으로 확인.

## 검증 3 — 비용 최적화가 판단을 흔들지 않는가 : 구조 PASS, 단 계측 미연결

- **캐싱(C-2) — 판단 불변(구조 보장):** cache_control 은 과금·지연만 바꾸고 샘플링 분포를 안 바꾼다
  (Anthropic 프롬프트 캐싱 성질). 로컬 판단 캐시 키 `_cache_key(model,system,payload)`는 cache_control·
  schema 를 포함하지 않아 안정. → 캐싱이 심각도·방향·계정·근거·주체성·감사중요를 바꿀 수 없다.
- **출력 다이어트(C-1) — 근거 불변, 화면 미사용만 제거:** 제거 4필드(relevance_reason·direction_reason·
  intrinsic_risk_reason·magnitude.reason)는 grep 상 화면(judgment_view·render·index.html) 어디에도 없음
  (오직 serialize `_judgment_from_dict` 덤프복원 경로에만). 근거(evidence·amount_quote·계정 quote·
  denominator_reason)·표시이유(one_line_reason)·모든 축 결론값 유지. → **원문 근거 안 줄었다(원칙 3·5).**
- ★**계측(C-3) 결함 — 실사용에서 안 남음:** `tools/cost_meter.py`(CostMeter)는 단위상 정확하나, **어떤
  진입점(CLI `pipeline.main`·Flask `server.py`)도 CostMeter 를 인스턴스화하거나 `on_usage`를 run()에
  전달하지 않는다**(repo 전수 grep: 정의 파일 밖 사용처 0). 실행 시 on_usage=None → **비용 로그가 실제로
  남지 않는다.** changelog 의 비용 수치는 개발방의 임시 계측 실행에서 나온 것이지 배선된 계측이 아니다.
  → 처방 C.
- ★**라이브 필요(미검):** 유지된 판단값들이 C-1(schema)·system.txt(간결 지시) 변경 후에도 동일한지는
  판단 캐시가 전량 무효화돼 라이브 Opus 필요. 구조상 모든 판단 필드는 스키마에 남아 있음(값 동일성만 미검).

## 처방 (harness/pending/)

### A. [검증방 라이브] prescreen 국면 분할 '실거동' 불변식 — 처방 §B 잔여
결정론 코드 메커니즘은 위 신규 불변식이 고정했다. 남은 것은 **실제 Haiku 가 5국면에 서로 다른 event_id
를 매기는지**(§B 원안: red→green, 실 Haiku). 사용자 고지 후 소량 라이브로: 담합 5국면(심의/첫재판/과징금
확정/집단소송/정책검토)을 실제 prescreen 에 태워 '최소한 과징금확정류가 survivors 에 남는다'를 확인.
대조군 부산/울산 미병합 유지. 비용 억제(같은 국면 다매체 접힘)도 함께 관측.

### B. [개발방] 최신성 프로브의 '베스트 에포트'가 네트워크 예외를 안 잡는다 (§2 견고성)
`_statement_rows`(dart.py:220-223)가 `except (DartRateLimited, DartConfigError)`만 잡는다. 최신성 프로브
중 `requests.exceptions.ConnectionError/Timeout`(DART 호스트 다운·타임아웃)이 나면 **미포착 → 전파 →
run 전체 크래시.** changelog 는 "네트워크 막히면 캐시 유지, 크래시 금지"라 했으나 코드가 그걸 보장하지
못한다(청구-코드 불일치). 루프14 이전엔 캐시 적중 시 네트워크 0콜이라 이 위험이 없었고, 2026년 내내
모든 캐시가 latest(2026) 기준 stale 이라 프로브가 매 실행 발동 → 상시 노출. **로우드(크래시=명시적)라
§1 침묵 증발은 아니나**, 값싼 수정 권고: except 에 `requests.exceptions.RequestException`(또는 베스트
에포트 구간을 넓은 except 로) 추가해 프로브 실패 시 캐시 유지로 하강.

### C. [개발방] 비용 계측이 실행 경로에 미연결 — '실제로 남지 않음'
`CostMeter`를 진입점에 배선한다: `pipeline.main`(및 원하면 server 분석 경로)에서 `meter=CostMeter()`를
만들어 `run(..., on_usage=meter.on_usage)`로 넘기고 종료 시 `meter.report()`를 출력/로그. 배선 전까지는
"계측은 개발용 도구이며 기본 실행에선 집계 안 됨"을 정직히 표기(원칙 4·5 — 안 남는 걸 남는 척 금지).

### (참고, 처방 아님) 경미: serialize `_judgment_from_dict`가 C-1 제거필드를 subscript(`d["..."]`)로 읽음
parse_batch 는 `.get` 방어인데 복원부는 strict. 현 예시 덤프 2개는 키를 다 보유(asdict 가 dataclass 필드를
항상 채움)라 실害 없음. 방어 일관성 위해 복원부도 `.get(...,"")` 권장(선택).

## release-criteria 대조
- B1(국면 오병합) = §1 '서로 다른 사건의 오병합'. 코드 메커니즘·역할분리로 해소 확인, 라이브 거동만 잔여(A).
- B2(분모 staleness) = §1 '재무 분모가 최신 보고서가 아닌 것'. 동적 `_years()`+무효화로 해소, 제19기 수신
  읽기전용 실증. **미해결 blocker 목록에서 내려도 되는 근거 확보**(단 end-to-end 캐시 쓰기는 라이브 잔여).
- 처방 B·C 는 §1 침묵 결함이 아니라 §2(견고성·계측). 출시를 막지 않으나 값싸게 고칠 것.

_비개발자 요약:_ 두 핵심 문제가 실제로 고쳐졌는지 직접 확인했습니다. (1) 담합 사건의 여러 국면(조사→재판→
과징금 확정→소송)이 앞단 정리기에 삼켜지지 않고 각각 살아남도록 코드가 바뀐 걸 무료 재현으로 확인하고,
되돌아가지 못하게 자동 검사(불변식)를 새로 걸었습니다. (2) 재무 규모의 분모가 '올해 기준 최신 보고서'를
자동으로 찾도록 바뀐 걸, 실제 DART에서 CJ의 최신 제19기(자산총계 29.7조)를 읽어와 감사인이 말한 숫자와
똑같음을 확인했습니다(기존 데이터는 건드리지 않음). 비용 절감은 판단 결과를 바꾸지 않음을 구조로 확인했으나,
비용을 '기록'하는 계측기가 실제 실행에는 연결돼 있지 않아 지금은 로그가 안 남습니다 — 그 배선과, 인터넷이
잠깐 끊길 때 프로그램이 죽지 않게 하는 보완을 처방으로 남겼습니다. 실제 AI 판단 품질(유료 호출)은 이번에
비용 없이 확인 불가라 명확히 분리해 남겼습니다.
