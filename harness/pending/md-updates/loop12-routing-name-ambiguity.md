# 관측(설계자·감사인 판단 요청): 동명 계정의 라우팅 잔여 모호성 — ID 정밀화는 '가짜 정밀'

- 발의: 검증방 / 루프12 검증(계정 증발이 진짜 막혔나) / 2026-07-20
- 성격: **코드 결함 아님.** 선택/그룹 무결성은 안정 키로 해결됨(루프12 검증 통과). 이 항목은 '라우팅
  의미 단위=이름'에서 오는 구조적 잔여 모호성의 정책 판단 + UX 처방.
- 상태: **개발방 루프13 반영 완료(UX 안내 한 줄). 라우팅 로직 불변.**

## ▶ 개발방 반영 보고 (루프13 / 2026-07-20)
처방의 '표시 UX만' 부분을 반영했다: 계정 선택 지점(index.html 계정 패널)에 안내 한 줄 추가 —
"배정은 계정군(이름) 단위입니다 — 유동/비유동처럼 이름이 같은 계정을 고르면 같은 이름의 뉴스가
함께 배정됩니다(더 보이는 쪽이라 안전, 정밀 확인은 카드의 계정 가설·근거로)." 라우팅 로직·판단은
불변(selectedRouteNames→이름→classify_and_rank 이름 매칭 그대로). 처방의 '하지 말 것'(ID 라우팅
정밀화=가짜 정밀, 원칙 4) 준수 — ID 라우팅으로 바꾸지 않았다. 고정표·키워드 없음(원칙 1).

## 무엇을 관측했나

루프12는 선택·그룹 토글을 '안정 키(행 위치)'로 정밀화했다(동명 계정을 정확히 구분해 켬 — 검증 완료).
그러나 **라우팅으로 보내는 값은 여전히 '이름'**이다(`selectedRouteNames()`, index.html:418 — 선택된
말단의 표시명을 dedup 해 /api/route 로 전달). 그래서:

- CJ에서 **비유동 계약부채(key45)만** 골라도 라우팅에 가는 값은 이름 "계약부채"다. 뉴스가 계정에
  붙을 때 LLM은 이름("계약부채")으로만 태깅했지 유동/비유동을 **구분한 적이 없다**. 결과: 비유동만
  골라도 "계약부채"로 태깅된 뉴스가 (유동 관련이든) 그 바구니(①)로 들어온다 → **과다포함**.

## 판정 (검증방 의견)

1. **원칙 7(누락 금지) 관점: 안전한 방향이다.** 과다포함은 '더 보임'이지 '덜 보임'이 아니다. 감사
   도구는 거짓양성(과다포함) < 거짓음성(놓침)이 원칙(principles 7·journey). 방향이 옳다.
2. **감사인 혼란 가능성: 낮으나 존재.** '비유동만 골랐는데 왜 이게?' 할 수 있다. 완화책은 이미 일부
   있다 — 선택 시점에 context 라벨(유동부채/비유동부채)로 무엇을 골랐는지 드러나고, 뉴스 카드에
   계정 가설+근거가 붙어 감사인이 5초에 판단 가능(원칙 3·5).
3. **★ ID 라우팅으로 '정밀화'하는 것은 가짜 정밀(원칙 4 위반)이다.** LLM의 계정 연결은 계정군(이름)
   수준이다(account-linkage.md D7 — "정확 계정명 매칭을 강제하면 회사마다 깨지고 특정회사 맞춤으로
   샌다. 연결은 '이 근처 계정군' 수준으로 느슨히, 정밀 매칭은 사람이 한다"). 뉴스 태그에는 account_id
   가 없다(이름 문자열뿐). 선택된 ID와 태그를 ID로 매칭하려면 LLM이 유동/비유동을 구분해 태깅해야
   하는데, 얇은 헤드라인으로 그걸 확신시키면 **LLM이 구분한 적 없는 것을 구분한 척하는 날조**다.
   → **라우팅은 이름(계정군) 수준으로 두는 것이 옳다.** 개발방이 '선택/그룹 무결성=안정 키, 라우팅
   의미=이름'으로 층을 나눈 것은 원칙에 부합한다.

## 처방 (표시 UX만 — 판단·라우팅 로직 불변)

- **화면 안내 한 줄 추가**: 계정 선택/적용 지점에, "라우팅은 계정군(이름) 단위입니다 — 유동/비유동처럼
  이름이 같은 계정을 고르면 같은 이름의 뉴스가 함께 배정됩니다(정밀 확인은 카드 근거로)"를 노출.
  이는 느슨함을 숨기지 않는 것(원칙 7)이지 로직 변경이 아니다. 고정표·키워드 없음(원칙 1).
- **하지 말 것**: 라우팅을 account_id 로 바꾸기(가짜 정밀 — 위 판정 3). 동명 계정을 '유동/비유동'
  고정 라벨로 강제 분해해 라우팅 키로 쓰기(원칙 1·4 위반).

## 라이브 미검(정직 고지)
- '비유동만 골랐을 때 유동 계약부채 뉴스가 실제로 들어오는지'의 라이브 실증은 판단(LLM 계정 태그)이
  필요해 API 키 없는 이 환경에서 미실행. 위 판정은 코드 구조(selectedRouteNames→이름→classify_and_rank
  이름 매칭)와 account-linkage.md 원칙에서 도출. 메커니즘은 결정론적으로 확인됨.

_비개발자 요약:_ 이제 화면에서 유동/비유동 계약부채를 정확히 구분해 고를 수 있습니다. 다만 '뉴스를
어느 담당자에게 보낼지'는 계정 '이름' 기준이라, 비유동만 골라도 이름이 같은 '계약부채' 뉴스가 함께
갑니다. 이건 덜 보이는 게 아니라 더 보이는 쪽이라 감사엔 안전한 방향이고, 이름을 무리하게 ID로
쪼개 정밀한 척하면 오히려 AI가 하지도 않은 구분을 지어내는 셈이라 권하지 않습니다. 대신 '이름
단위로 배정된다'는 안내 한 줄을 화면에 두는 걸 제안합니다.
