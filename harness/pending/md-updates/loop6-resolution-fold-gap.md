# 처방(발의): 비법적 '위험 해소'가 '단순 호재'로 접히는 표시 공백

- 발의: 검증방 / 루프6 검증 / 2026-07-17
- 우선순위: 낮음(악재 거짓 탈락 아님 — 해소 신호의 저평가. 접힘=접근 가능)
- 상태: pending (판단층 루프에서 검토 — 표시-only 스코프 밖)

## 발견 (검증 4 — D20 이 검증방에 넘긴 공백을 형식화)
`serialize.default_visible` 은 '위험 해소'를 `stage=='무혐의/취하'` 로만 읽는다(D20).
그래서 **법적 절차가 아닌 위험 해소**(예: 워크아웃 졸업, 채무재조정 완료, 유동성 위기
해소, 신용등급 상향 복귀)는 stage 축에 안 담겨 `direction=='호재'` 로만 판정되고,
default_visible 이 이를 '단순 호재(위험 해소 아님)'로 **접는다**.

- 이는 악재 거짓 탈락(원칙 7 핵심)은 아니다 — 접힌 항목은 folded_view 로 보존·접근 가능,
  게다가 좋은 소식이다. 그러나 routing-rules(루프5)가 '위험 해소는 기본 화면 노출'로
  의도한 바에 비해, 비법적 해소는 기본 화면에서 빠진다(해소 신호의 저평가).
- 실측(루프6): 법적 해소(검찰 '혐의없음')는 stage=무혐의/취하로 정상 노출됨(통과). 비법적
  해소는 이번 케이스셋에 없어 미발현이나, 코드 경로상 접힘이 확정적이다.

## 처방 후보 (판단층 루프 — 표시만으로는 못 고침)
- (a) 판단 스키마에 '위험 해소' 개념을 stage 너머로 일급화(예: resolution:bool + 근거).
  그러면 default_visible 이 stage 문자열에 의존하지 않고 판단값을 읽어 노출한다.
  주의: 키워드로 '졸업/해소' 단어를 잡으면 원칙 1 위반 — LLM 이해로 판정해야 한다.
- (b) 그대로 두되 문서화: 비법적 해소는 '단순 호재'로 접히고 접힘 줄에서 접근 가능함을
  routing-rules 에 명시(현 동작을 의도로 승인). 가장 가벼움.
어느 쪽이든 '악재'는 계속 노출되므로 이번 루프 핵심(거짓 탈락)과는 무관하다.

## 근거 위치

- decision-log D20(남은 공백), docs/routing-rules.md(루프5 기본 화면 표시 규칙)
