## 검증방이 루프11에서 확인한 것 (요약)
- **BS 섹션 수정은 일반 해법(원칙 1·2 CLEAN).** `_BS_SECTION_SUBTOTAL_IDS`는 표준 IFRS 택소노미
  ID(ifrs-full_*) 9개뿐 — 한국어 리터럴·회사코드·CJ 특정 계정('매각예정 처분집단 부채'는 집합에
  없음, 산술이 잡는 leaf) 없음. 판정 경로는 ID 동치 비교이고 한국어는 출력 섹션 라벨일 뿐이다.
  · **실측(무-API):** 앵커는 CJ(00635134)에서만 2행(부채총계·grand)을 실제로 보완했고 나머지
    4사(삼성·현대차 등)에선 is_total==is_sub 로 **미작동(inert)** → 개발방 '4사 바이트 동일'은
    '일반성'이 아니라 '그 4사엔 앵커가 아무것도 안 함'을 뜻한다(구분해 보고).
  · **일반성은 반례로 입증:** (A) 실물 CJ red→green — 앵커를 끄면 부채 0·미확정 21로 무너지고
    켜면 부채 21·미확정 0(앵커 load-bearing). (B) 삼성(다른 회사)에 같은 병리(매각예정 비0)를
    주입해도 앵커가 똑같이 고침 = CJ 바이트가 아니라 '메커니즘'이 고침. (C) 섹션 블록 순서 3종
    재배열에도 동일 배정. → CJ 맞춤 아님.
- **미확정 폴백 CLEAN(원칙 4·7).** 구조로 못 놓은 행은 인접 섹션에 안 밀리고 None(미확정)으로
  분리, fallback_used=True, 화면 '구분 미확정(확인 요망)' 별도 섹션 + 안내문. 그룹선택(루프8)
  소속에도 안 섞임.
- **감사중요(audit_attention) CLEAN(원칙 8·3·4·1).** severity_rank·정렬 불변, 표시층만 값을 읽음,

## 검증 못 한 것 / 경계 (정직 — 원칙 4)
1. **BS 앵커 집합의 완전성 경계(일반성 한계, 결함 아님).** 집합은 IFRS SoFP 의 '표준 9개' 구간합계
   만 담는다. IFRS 는 이 밖의 표시용 소계(예: 매각예정 처분집단 '자산' 총계가 소계로 표시되는 변형,
   드문 표시관행)를 허용한다. 그런 소계를 산술이 놓치고 집합에도 없으면 그 섹션이 불완전해질 수
   있다. **단 안전하게 열화한다** — 인접 섹션 오염이 아니라 '미확정'으로 빠지고 fallback_used=True
   로 화면에 정직히 드러난다(원칙 4·7). 즉 '틀리게 배정'이 아니라 '못 배정을 표시'.
2. **라이브 정확도(크레딧 필요).** 프롬프트 3곳 변경으로 기존 판단캐시가 무효 — 'LLM 이 audit_
   attention 을 과다하게 켜지 않는가', '방향 정밀화가 실제 기사에서 맞는가', '중복이 실제로 옳게
   묶이는가(드리프트 있는 실데이터)'는 라이브가 필요하다. 이 환경엔 크레딧 없어 미검(억지 통과
   주장 안 함). 표시·BS·정렬 메커니즘은 결정론적으로 실증됨.

## 처방
1. **(카나리 — 검증방/개발방) BS 미확정 발생을 라이브에서 관측·기록.** fallback_used=True/
   unassigned>0 는 이미 화면에 뜨지만, 회귀검증이 '어느 실회사에서 미확정이 났는지'를 남기도록
   (loop7-bs-fallback 카나리 확장). 미확정이 잦은 표시관행이 발견되면 그 표준 IFRS 소계 ID 를
   집합에 '표준 마커로서' 추가할지 설계자 판단(이름표가 아니라 ID 추가는 원칙 1 위반 아님).
2. **(설계자 판단) 앵커 집합을 '고정 9개'로 둘지, IFRS SoFP 소계 네임스페이스에서 유도할지.**
   고정은 단순·안전(못 잡으면 미확정)하나 커버리지 경계가 있고, 유도는 커버리지↑지만 과잉표시
   위험. 트레이드오프 — 개발방 자의 금지(원칙 10).
3. **(재현성 — pending #1 재확인) CJ 를 커밋 예시 덤프로.** CJ 는 비정형 BS + 감사중요 + 매각예정을
   모두 가진 대표 병리 케이스다. out/results 에 커밋해 두면 다음 검증방이 BS/감사중요/라우팅을
   라이브 없이($0) 재검할 수 있다(total-verify-observations #1 과 같은 방향).

_비개발자 요약:_ 재무상태표를 자산/부채/자본으로 나누는 이번 수정이 'CJ 한 회사에만 맞춘 편법'이
아니라 '모든 IFRS 회사에 통하는 표준 방식'임을, 다른 회사(삼성) 데이터에 같은 문제를 일부러 심어도
똑같이 고쳐지는지 실험해 확인했습니다. '남의 기사 언급을 잘못 세는' 노이즈를 줄이려 더한 '감사
확인 요망' 표시도, 근거 없는 판단을 몰래 끌어올리지 않고 순위를 건드리지 않음을 확인했습니다.
다만 (1) 아주 드문 표시 방식의 회사에선 일부 계정이 '구분 미확정'으로 빠질 수 있는데 이는 틀리게
넣는 게 아니라 정직히 '확인 요망'으로 남기는 것이고, (2) 'AI 가 실제로 판단을 잘 하는가'는 유료
API 가 있어야 확인할 수 있어 다음 과제로 남깁니다.
