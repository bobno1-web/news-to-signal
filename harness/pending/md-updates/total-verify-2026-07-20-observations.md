## 1. [핵심 통과] 화면 계정명 = 회사 보고 명칭(account_nm), 우리 라벨 아님 — §1 통과
사용자(감사인) 의혹("기타의 비유동자산·총 기타유동금융부채 등이 표준 라벨처럼 보인다")을
DART 원본으로 전수 추적해 반증했다(CJ 51 + 현대차 57 + 태영 47 표시행 전부, 화면표시명 ==
원본 account_nm, 불일치 0건).
- ★결정적 반증: **같은 IFRS 표준 태그(account_id)가 회사마다 다른 한국어 이름**을 낸다:
  - `ifrs-full_OtherCurrentAssets` → CJ "기타의 유동자산" / 현대차 "기타자산" / 태영 "기타유동자산"
  - `ifrs-full_OtherCurrentFinancialLiabilities` → CJ "총 기타유동금융부채" / 현대차 "기타금융부채" / 태영 "기타유동금융부채"
  - `dart_ElementsOfOtherStockholdersEquity` → 현대차 "기타자본항목" / 태영 "기타자본구성요소"
  우리가 태그로 라벨을 붙였다면 같은 태그는 같은 이름이 나와야 한다. 다르다 = 이름은 각 회사의
  account_nm 이다. 코드도 account_id 를 '읽기만' 하고(합계/섹션 탐지 내부용) 화면 이름으로 절대
  안 쓴다(dart.py). raw 캐시는 DART fnlttSinglAcntAll 응답 그대로(가공 0). → 울타리 원칙 지켜짐.
- 부수 확인: screen-vs-raw 및 fallback 계정보존 불변식이 캐시 5사에서 실제로 돌아 통과(증발 0).

## 3. [관측 §2 — blocker 아님] B-4 자기모순 조항이 약함 (판단층 처방)
- B-4(severity_rank 자기일관성) 조항은 프롬프트에 실재한다(system.txt: severity_rank 항목 "자기
  일관성(★)"). 표시층 문자열 필터가 아니라 판단층 서술이라 원칙 1 준수. 그런데 약하게 작동:
  - (a) 자기축소 **형용사 열거**(경미·소규모·소액·국지적·일회성)에 묶여, LLM 이 다른 표현으로
    쓰면("업계 공통 이슈로 회사 특정 사건 아님", "영향 제한적") 조항이 안 걸린다 — 원리가 아니라
    단어에 기댄 미세한 원칙1 냄새.
  - (b) 트리거 단어가 있어도("소규모 원가 영향") LLM 은 사건 전체가 아니라 **하위 측면(원가 영향)**만
    축소한 것이라 '내가 사건을 경미하다 판단'으로 안 읽는다.
  - (c) severity_rank 는 홀리스틱이라 다른 축(200억 절대금액·본질위험)이 상쇄해 ★자기일관성 넛지를 이긴다.
- 처방(판단층·원칙1, 표시층 문자열 필터 금지): 조항을 단어열거가 아니라 **원리**로 — "one_line_reason
  을 다시 읽어라. 재무제표 영향이 제한적/업계공통/회사 비특정이라 서술했다면 상단 밴드에 두지 마라"의
  자가 사전점검(emit 전). '회사 특정 사건 아님'은 audit_attention 의 '회사 특정성 관문'과도 연결 —
  이 회사 F/S 에 안 닿는다 판단했으면 이 회사 레이더 상단이 아니다. 라이브 재검은 §2라 출시 필수 아님.
- release-criteria §2 가 "이유에 '소규모'라 쓰고 상단 노출"을 명시적 **비-blocker(관측)**로 분류 —
  근거(이유+위치)가 화면에 함께 보여 감사인이 스스로 바로잡는다. 출시를 막지 않는다.

## 4. [기존 blocker 재확인 — 미해결] 규모 분모 기수 staleness (§1)
- `_YEARS=["2024","2023","2022"]` 하드코딩 → FY2025(제19기) 절대 미조회. 캐시(statement_*.json)는
  무효화 정책이 없어 제18기(FY2024)에 영구 고정. CJ 화면 자산총계 30.15조(제18기) vs 사용자 제시
  제19기 29.7조 = ~1.5% 차. 코드는 당기(thstrm)를 정확히 읽음 — '컬럼 오인'이 아니라 '낡은 보고서'다.
- 영향: 지금은 작다(비율은 앵커로만 표시, size 를 코드가 임계 확정 안 함 — 원칙8). 그러나 (a)
  시간이 갈수록 커지고 (b) 화면의 "30.1조"가 최신 공시와 어긋나 감사인 신뢰를 깎으며(발단 의혹과
  같은 결) (c) YoY 변동 큰 회사는 분모가 크게 틀린다. release-criteria §1('재무 분모가 최신
  보고서가 아닌 것')에 정확히 해당 → blocker. 이미 urgent-account-provenance #4 + 부록에 등록됨.
- 처방: `_YEARS` 를 현재연도부터 역순으로 **동적 계산**(하드코딩 제거) + 캐시 무효화/갱신(최신
  사업보고서 rcept_no·기수를 조회해 캐시보다 새로우면 갱신). 특정 회사·특정 보고서 하드코딩 금지.
  같은 사안이 매출·자기자본 분모에도 적용. (개발방 코드 루프; '최신 판별 기준'은 설계자 정책.)
