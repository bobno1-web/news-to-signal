# 처방(발의): '넓은 1년 수집'의 실제 상한(네이버 start≤1000)을 문서·표기로 정직화

- 발의: 검증방 / 루프6b 검증 / 2026-07-17
- 우선순위: 낮음(원칙 7 위반 아님 — 이미 표기됨. 완전성 한계의 추적)
- 상태: pending

## 발견 (검증 1 실측)
태영건설 넓은 수집을 직접 돌린 결과 collected=1056, meta.hit_naver_ceiling=**true**
(raw 1300 = date 1000 + sim 300, URL중복 244 제거). 즉 뉴스가 아주 많은 회사는 네이버
검색 API 의 start≤1000 하드캡 때문에 '최근 1년 전체'가 아니라 '최신순 ~1000 + 정확도순
300'만 잡힌다. date_pages_items=1000 이 상한에 정확히 걸렸다.

- 이는 숨김이 아니다: `last_broad_meta.hit_naver_ceiling` 로 표기되고 UI 도 '1000 상한
  도달'을 보여준다(원칙 7 준수). 그러나 '1년치 넓은 수집'이라는 표현은 고volume 회사에서
  실제로는 '최근분 상한 + 관련도 보강'임을 사용자가 오해할 수 있다.
- 위험 관점: 상한에 걸리면 1년 창의 '오래된 꼬리'(예: 8~12개월 전 사건)가 date 경로로는
  안 잡힐 수 있다. sim 경로(정확도순 300)가 일부 보강하나 완전하지 않다.

## 처방 후보 (개발방/설계 판단)
- (a) 문구 정직화: UI·README·docs 에 '네이버 API 상한(≤1000)으로 고volume 회사는 최근분
  중심 + 관련도 보강'임을 명시(가장 가벼움, 원칙 7 정합 강화).
- (b) 커버리지 개선(선택): 네이버가 날짜범위 파라미터를 안 주므로, 회사명 + 기간 힌트를
  나눈 복수 쿼리로 창을 쪼개 보강하는 방안 검토. 단 쿼리에 '부정 키워드'를 넣지 말 것
  (원칙 1) — 기간/페이지 축만 쪼갠다.
- 판단 영향 없음: 이 한계는 '수집 완전성'이며 7축 판단·깔때기 안전성과는 무관하다.
