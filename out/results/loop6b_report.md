# 루프6b 완료 보고 — 수집 제한 제거 + 넓은 수집 실제 반영

## 1. 1단계 진단 (경우 A/B, 실제 수집 건수)

**결론: 경우 B — 백엔드 미구현.** UI만 제한한 게 아니라, '1년치 수집 + 3단 깔때기'
자체가 코드에 없었다.

- 진단 근거(고치기 전 코드):
  - `NaverNewsSource.fetch(display=30)` = **단일 요청 1페이지**. 페이지네이션·기간창·
    정확도+최신 병행 수집 없음. 네이버 1요청 최대 100건에 갇힘.
  - `pipeline.run(display=30)` = 수집 → `judge_items`(Opus 7축) → `classify_and_rank`.
    그 사이에 **중복제거·Haiku 스크리닝·3단 깔때기 함수가 존재하지 않았다.**
  - `server /api/analyze` 가 화면 드롭다운의 `display`(15/30/50)를 그대로 이 단일
    요청에 넘겼다. 그래서 태영건설이 30건(최신 단일 요청)에 갇혀, 그 30건이
    워크아웃/PF 도배로 채워졌다.
- 이는 루프6(검증방)이 이미 '지시가 가정한 스크리닝 게이트·중복제거가 코드에 없다'고
  전제 불일치로 지적한 바와 정확히 일치한다(그 처방이 이번에 설계 결정으로 회수됨).
- **고친 코드에서의 실제 수집 건수(태영건설): 1,056건**
  (네이버 최신순 1000건 상한 도달 + 정확도순 300건, 동일 URL 중복 244건 제거,
  최근 1년 내 1,056건. 태영은 뉴스가 조밀해 최신 1000건이 1년을 다 못 덮어 상한에
  걸림 — 화면에 '1000건 상한 도달' 표기).

## 2. 수정 내용

- **수집 건수 드롭다운(15/30/50) 제거.** 사용자는 수집 개수를 고르지 않는다.
  수집은 항상 시스템 기본값으로 최근 ~1년치를 넓게 자동 수행한다
  (`NaverNewsSource.fetch_broad`: 회사명 단일 검색어, 정확도순+최신순, 페이지네이션,
  1년 필터, 동일 URL 중복만 literal 제거). 네이버는 날짜범위 파라미터가 없어
  '기간 창 분할'을 직접 못 하므로 최신순 페이지네이션+클라이언트 필터로 근사하고,
  1000건 상한 도달은 숨기지 않고 화면에 표기한다(원칙 7).
- **3단 깔때기 실제 구현**(`src/judge/prescreen.py`, Haiku): 넓게 모은 뒤 판단 '앞'에서
  ① 중복 정리(같은 사건을 한 대표로) → ② 간이 관련성 관문(명백 무관만 접음) →
  ③ 대표만 기존 Opus 7축 풀판단. 대규모(1000+건) 대응으로 2패스 군집
  (배치별 군집 → 대표만 모아 전역 재군집).
- **화면 상단에 '수집 N · 표시 M · 접힘 K' + 깔때기 흐름줄** 표시(넓게 수집됐음이 보이게).
- **삭제 없음(원칙 7):** 중복·간이무관도 전부 하단 접힘 줄에 보존(펼치면 전부 접근).
  정합: 수집 N = 표시 + 접힘(판단후 + 입구).
- **판단(7축) 로직·스키마·프롬프트·라우팅은 불변** — 깔때기는 입구다. 비용 측정을 위한
  순수 텔레메트리(on_usage, 판단 입출력 무영향)만 engine 에 추가.

### 원칙 준수(역검증)
- 원칙 1: 같은 사건인지·무관인지는 Haiku 가 '기사를 이해'해 판정(event_id·relevant).
  코드는 그 결과를 묶기만 — 키워드·문자열 유사도 임계값 없음. 병합은 보수적(애매하면
  안 합침 — 다른 사건 병합=하나 증발 금지), 관문은 느슨(애매하면 통과 — 놓침>오탐).
- 원칙 2·9: 수집은 벤더 어댑터 뒤(fetch_broad), 코어는 인터페이스만. src→reference 없음.
- 원칙 10: 약한 모델이 앞단에서 거르는 '거짓 탈락'의 새 표면은 검증방이 불변식으로
  확인해야 한다 — pending 두 건(screening-premise / dedup-must-not-merge)에 반영·이관.

## 3. 태영건설 재실행 — 단계별 실제 건수 (라이브 실측)

```
수집 N = 1,056
  → (중복 정리)      중복 766건 접힘 → 남은 distinct 290
  → (간이 스크리닝)   명백 무관 142건 접힘
  → 풀판단 J = 148   (Opus 7축 판단 대상)
      → 그중 기본 화면 악재 34건 · 판단후 접힘 114건(단순호재 36·무관 60·중립 18)
```
- 중복 정리로 워크아웃/PF 도배(같은 사건 수백 건)가 대표 1건씩으로 접히고, 그 덕에
  다른 사건이 판단 무대에 올랐다.
- **정합 확인(증발 없음): 수집 1,056 = 표시 34 + 판단후접힘 114 + 깔때기접힘 908
  (중복 766 + 간이무관 142) = 1,056 → OK.** 어느 것도 삭제되지 않고 전부 접근 가능.
- (수집은 라이브 네이버라 실행마다 총량이 조금씩 달라진다 — 위 숫자는 이번 실행 실측치.)

### 화면 악재 목록에 워크아웃/PF 외 새로 들어온 악재 (실물)
기존 30건 화면에서는 최신 워크아웃/PF 반복 기사만 보였다. 넓은 수집 + 깔때기 뒤에는
같은 워크아웃 사건이 대표 1건으로 접히고, 아래처럼 **성격이 다른 감사 관련 신호가 함께**
기본 화면에 올라왔다(실물 발췌):

- **계열사 채무 담보 제공(특수관계자·우발채무):** "태영건설, 계열사 네오시티 채무 담보로
  주식 138억원 담보제공 결정" → 특수관계자 거래·우발채무 확인 요망.
- **정비사업 계약 해지(매출 직격):** "태영건설, 부산 명보아파트 가로주택정비사업 계약
  해지" → 매출 약 2.7% 규모 계약 해지.
- **PF 사업장 공매 처분(자산손상):** "DS단석, 태영건설 '반포 PF 사업장' 공매 낙찰" →
  손실·계속기업 영향 확인 요망.
- **하도급대금 지급여력:** "건설사 하도급대금 지급 '온도차'…10일 내 지급률 0.75~92.09%"
  → 유동성·지급여력 신호.
- **노무 우발채무:** "중견 건설사 '사용자성' 속속 인정…서희·코오롱·태영·금호…" →
  사용자성 인정에 따른 노무 관련 우발채무 가능성.
- **매출원가 압박:** "건설업계 철근 매입가 일제히 상승" → 원가 부담(금액 불명은 미상 유지).
- **재무구조 취약(구체 수치):** "금호 551%·태영 489%…중견건설사 부채비율 살얼음" →
  부채비율 489% 직접 노출.
- **지배구조·통제환경:** "태영건설 지배구조 지표 준수율 46%…CEO 승계체계 미비".

즉 '워크아웃'이라는 한 사건에 갇히지 않고, **특수관계자 담보·계약 해지·자산 공매·
하도급 지급여력·노무 우발채무·원가·부채비율·지배구조** 등 감사인이 볼 서로 다른
축의 신호가 드러났다. 넓게 모으지 않았다면 30건 도배에 묻혔을 것들이다.

## 4. 비용 실측 (Haiku · Opus 토큰, USD)

| 모델 | 용도 | 호출 | 입력 토큰 | 출력 토큰 | 비용 |
|---|---|---|---|---|---|
| claude-haiku-4-5 | 깔때기(중복정리+간이관문) | 10 (콜드) | 225,019 | 25,472 | **$0.35** |
| claude-opus-4-8 | 7축 풀판단(148건, 10배치) | 10 | 147,053 | 91,761 | **$3.03** |
| **합계** | | | | | **≈ $3.38** |

- 단가: Haiku $1/$5, Opus $5/$25 (입력/출력, 1M 토큰당).
- Opus 출력 91,761토큰에는 adaptive thinking 토큰이 포함된다(과금 대상). Opus 148건은
  전부 신선 측정(캐시 삭제 후). Haiku는 1,056건 콜드 프리스크린 실측치($0.35)를 싣는다
  (이번 재실행은 프리스크린 캐시 재사용으로 신규분 $0.11만 발생 — 콜드 기준이 대표값).
- **압축 효과:** 깔때기로 Opus 풀판단을 1,056 → 148건(약 1/7)으로 줄였다. 만약 Opus로
  1,056건을 전부 판단했다면 대략 $21.6(≈$3.03×1056/148)가 들었을 것을, Haiku $0.35를
  더해 $3.38로 처리했다. 즉 **약 6배 이상 비용을 압축**했다(넓게 보면서도 저렴하게).

### 형식 외 관측(정직)
넓은 수집은 태영이 '비교 대상'으로만 언급된 타사 기사(예: 다른 건설사 부채비율 기사에서
태영 489%가 인용)까지 끌어오고, 판단(Opus)이 그 안의 태영 신호를 악재로 잡는 경우가
있다. 놓치지 않는 쪽(원칙 7)이라 타당하나, '태영이 주체가 아닌 지엽적 언급'과 '태영
자체 사건'을 화면에서 더 구분해줄지는 판단층/표시층의 다음 과제로 남긴다(이번 스코프는
입구). 또 약한 모델(Haiku) 관문이 진짜 악재를 떨구지 않는지는 검증방이 불변식으로
확인해야 한다(pending 이관).
