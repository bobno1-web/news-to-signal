# 방(개발방·검증방·설계방) 공통 규칙

## 세션 시작 필독 (예외 없음)
모든 방은 작업 시작 전 다음을 순서대로 읽는다:
1. CLAUDE.md
2. decisions/journey.md
3. docs/principles.md
읽지 않고 판단·구현을 시작하지 않는다.

## 개발방은 루프 시작 시 pending을 먼저 반영한다
개발방은 각 루프 시작 시 harness/pending/ 을 먼저 읽는다. 그 안의 MD수정·
새 불변식·훅을 해당 위치(docs/·tests/·.claude/)에 반영하고, 반영한 항목은
pending/에서 비운 뒤 harness/changelog.md 에 기록한다. 그 다음에 이번 루프
작업을 시작한다.

## 채점자·응시자 분리 (원칙 10)
개선의 발의권은 검증방(pending/에 처방), 시공권은 개발방(반영)이다. 개발방은
자신의 검증 기준을 스스로 바꾸지 않는다.

## 원칙 충돌 처리
- 이미 우선순위로 정해진 충돌 → principles.md 번호에 따라 자동 해소(낮은 번호 우선).
- 처음 보는 유형의 충돌 → 임의 해소하지 말고 harness/pending/ 에 올려 설계자
  판단을 받는다.

## 훅 강도 정책
자동 검사(훅)의 강도는 위반의 무게에 맞춘다:
- 차단(block): 정체성을 지키는 훅. 예 — src/ 가 reference/ 를 import 하면
  진행을 막는다(원칙 1·9의 물리적 방어선).
- 경고(warning): 습관을 챙기는 훅. 예 — 루프 종료 시 decisions/ 가 갱신되지
  않았으면 경고한다(막지는 않되 리마인드).
(훅의 실제 구현은 이후 루프에서. 이 문서는 정책만 못 박는다.)

## 산출물 설명은 비개발자 기준
모든 패치 결과·다음 패치 내용은 비개발자가 이해할 수 있게 한 문단으로 요약한다.
