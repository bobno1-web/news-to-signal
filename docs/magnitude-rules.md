# 규모 규칙 (magnitude-rules.md)

> 규모 축(severity-axes 축 2)의 상세 규칙. 규모는 다른 축과 성격이 다르다:
> 언어만으로는 풀리지 않고 숫자가 필요하다. 그래서 별도 문서로 다룬다.

## 규모는 두 신호로 분리한다
- 규모(magnitude): 크다 / 작다 / 미상 / 해당없음
- 금액 확실성(certainty): 확정 / 불명
  (※ "추정" 칸은 두지 않는다. "수백억대" 같은 범위는 정확 계산에 못 쓰므로
   불명으로 분류한다. 억지로 특정 숫자로 계산하면 그것은 날조다.)

## "미상"과 "해당없음"을 섞지 않는다 (표기 정합성)
- 미상: 재무·감사에 닿는 뉴스인데 금액 상한을 모른다 → 경보성("조사 중일 수 있음").
- 해당없음: 애초에 재무·감사와 무관한 뉴스(관련성 관문 탈락, relevant=false)라 규모
  자체가 의미 없다 → 규모 축을 비활성으로 표기("규모 해당없음").
무관한 마케팅 기사에 "미상(조사 중)"을 붙이면 경보 문구를 오용하는 것이다. 무관
뉴스는 "해당없음"으로 두어, "미상"의 경보 의미를 진짜 미상 사건에만 남긴다(원칙 5·6).

## "미상"은 "작음"이 아니다 (핵심)
금액이 안 잡히는 상황은 두 종류이며 정반대다.
- (A) 사소해서 금액이 없다(예: 로비 인테리어 교체) → 규모: 작음.
- (B) 아직 안 밝혀져서 금액이 없다(예: 진행 중인 횡령, 규모 조사 중)
  → 규모: 미상. 이는 낮은 심각도가 아니라, 상한을 모르기에 오히려 경보다.

시스템이 (B)를 (A)처럼 "금액 없음 = 작음"으로 처리하면 가장 위험한 사건을
바닥에 깐다. 이는 금지된다.

## 규모 판단의 세 경로
1. 금액 확정 → 회사 재무 규모 대비 비율로 규모를 판단.
2. 금액 미상 + 본질적 위험도 높음(부정 등) → 규모를 숫자로 계산하지 않고,
   다른 축(본질적 위험도·단계 등)이 심각도를 끌어올린다. 상단 근처에 둔다.
   화면에는 "규모 미상, 조사 중"으로 정직하게 표기한다.
3. 금액 없음 + 사소 → 하단.

## 분모 선택 (사건 성격에 따라)
규모 비율의 분모는 매출로 고정하지 않는다. 사건 성격에 맞는 기준을 쓴다.
- 매출 관련 사건 → 매출
- 횡령·자산손상 → 자산총계
- 유동성·자본 위기 → 자기자본
(구체 분모 매핑의 상세는 규모 루프를 실제 구현할 때 확정한다. 지금은
"매출 고정이 아니다"만 못 박는다.)

## 기권 규칙 (원칙 4)
금액이 명확할 때만 규모를 숫자로 판단한다. 모호하면 규모 판단을 기권하되,
심각도 판단 전체를 기권하는 것은 아니다. 규모 축만 빠지고 나머지 축이 그
뉴스의 심각도를 계속 매긴다.

## 참고: 재무 규모 데이터는 외부 소스(OpenDART)에서 온다
분모가 되는 회사 재무숫자는 DART 어댑터(src/sources/dart.py)를 통해 온다.
이 연결(뉴스 금액 ÷ 재무 규모)은 뉴스만 읽는 1차 판단과 성격이 다른 별도
작업이다. 특히 어려운 것은 DART 호출이 아니라 "뉴스에서 사건 금액을
신뢰성 있게 뽑는 것"이며, 뽑히지 않으면 위 경로 2/3으로 간다.

## 분모는 '연간 사업보고서'만 쓴다 — 반기·분기 금지 (설계 결정)
분모(매출액·자산총계·자기자본)는 **연간 사업보고서(reprt_code=11011)만** 사용한다.
반기·분기 보고서는 쓰지 않는다. 이유:
- 매출액은 '기간 누적' 값이다. 분기(3개월)나 반기(6개월) 매출을 분모로 쓰면 같은 사건
  금액이라도 비율이 과대하게 잡힌다(연 매출의 1/4~1/2 대비로 나눠지므로).
- 자산총계·자기자본은 '시점(결산일)' 값이다. 이를 '기간' 값인 분기 매출과 섞어 쓰면 분모끼리
  기준(시점 vs 기간)이 엇갈려, 세 분모가 서로 다른 회계 성격을 갖게 된다.
- 세 분모가 '같은 하나의 보고서·같은 기수'에서 나와야 비율끼리 비교 가능하다. 연간 보고서로
  고정하면 이 정합이 보장된다.

## 분모는 항상 '최신' 사업보고서여야 한다 (release-criteria §1)
분모가 낡은 보고서에서 오면 화면의 "자산총계 30조" 같은 수치가 최신 공시와 어긋나, 감사인이
근거만 보고는 낡음을 알아챌 수 없다(§1 blocker '재무 분모가 최신 보고서가 아닌 것'). 그래서:
- 조회 연도를 특정 값으로 하드코딩하지 않는다(원칙 1). 현재 연도부터 역순으로 동적 계산해,
  새 사업보고서(예: FY2025)가 공시되면 자동으로 잡힌다. 아직 공시 전인 최신 연도는 자동 탈락하고
  직전 연도로 내려간다.
- 캐시된 재무숫자도 무효화한다: 캐시의 사업연도가 지금 시점의 최신 후보보다 낡았으면, 캐시보다
  새로운 사업보고서가 공시됐는지 확인해(rcept_no 비교) 새로우면 갱신한다. 캐시가 특정 기수에
  영구 고정되지 않게 한다.
