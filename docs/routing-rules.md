# 라우팅 규칙 (routing-rules.md)

> 판단된 뉴스를 계정 담당자에게 어떻게 보여줄지 정의한다. 핵심: 계정 태그는
> "숨기는 게이트"가 아니라 "정렬하는 렌즈"다(원칙 7).

## 세 바구니 (계정 담당자 화면 기준)
1. 내 계정으로 의심되는 뉴스 — 위. 심각도순.
2. 계정이 특정되지 않은 실질 뉴스 — 가운데. 접힘. 단, 모두에게 보임.
   (기계가 계정을 못 붙였다는 것은 "관련 없음"이 아니라 "애매해서 사람이
   봐야 함"이다. 이것을 3번과 섞어 숨기면 안 된다.)
3. 명확히 남의 계정인 뉴스 — 아래. 접힘. 무시 가능.

## 정렬 규칙
- 먼저 바구니로 나눈다(누구 책상인가), 그 다음 각 바구니 "안에서" 심각도로
  정렬한다(무엇이 급한가).
- 심각도 정렬은 바구니를 가로지르지 않는다. 남의 계정 심각 뉴스가 내 계정
  뉴스보다 위로 올라오면 라우팅이 깨진다.

## 숨기지 않음 (원칙 7)
- 어떤 바구니도 뉴스를 완전히 제거하지 않는다. 낮은 것은 접되 접근 가능.
- 특히 2번(미배정) 바구니는 공용 큐다. 항상 접근 가능해야 한다.

## 양 조절은 라우팅이 아니라 임계치로
"뉴스가 너무 많다"는 계정 필터로 풀지 않는다. 심각도 임계치와 기간 창으로
조절한다. 라우팅(계정 렌즈)과 양 조절(임계치)은 별개의 손잡이다.

## 3층 노출 (원칙 5)
- 1층(항상): 제목 + 심각도 위치 + 계정 태그 + 한 줄 이유.
- 2층(펼침): 축별 근거 + 원문 근거 문장.
- 노출 안 함: 내부 점수·raw 계산값.
- 화면에는 등급이 아니라 "사람이 5초에 검증 가능한 이유"를 말로 표기한다.
  특히 "규모 미상"은 빈칸이 아니라 글자로 명시한다.

## 기본 화면 표시 규칙 = 악재 + 위험 해소 (루프5)
기본 화면(심각도순·계정별 공통)은 감사인의 '위험 레이더'다. 그래서 기본으로는
재무·감사 위험에 직접 관련된 것만 보인다:
- 노출: (1) 재무·감사 악재, (2) 위험 해소(리스크가 줄거나 사라진 신호).
- 접힘: (3) 단순 호재(재무에 닿지만 리스크 해소는 아닌 좋은 소식 — 수주·실적 등),
  (4) 재무·감사 무관, (5) 판단 기권(근거 부족).

이 분류는 새 고정표/키워드표가 아니라 **이미 내려진 판단값을 읽어** 배치만 정한다
(원칙 1·5·8 — 표시 계층은 판단을 재계산·변경하지 않는다):
- 악재 = direction=='악재'.
- 위험 해소 = stage=='무혐의/취하'. (우리 스키마엔 direction '해소' 값이 없다.
  프롬프트·severity-axes 상 리스크 해소는 direction 호재/중립 + stage 무혐의/취하로
  인코딩되므로, '해소'는 뉴스 단어가 아니라 stage(판단값)로 읽는다.)
- 무관 = relevant==False. 단순 호재 = relevant·호재·비해소. 기권 = abstained==True.

## 수집과 입구 깔때기 (루프6b)
수집 개수는 사용자가 고르지 않는다 — 시스템이 최근 약 1년치를 넓게 자동 수집한다
(회사명 단일 검색어, 정확도순+최신순, 페이지네이션). 대기업은 뉴스가 네이버 상한
(start≤1000)을 넘겨, 그 상한에 걸리면 화면에 '상한 도달'을 표기한다(숨기지 않음).
정직화(루프6b 검증 반영): 네이버 검색 API 는 날짜범위 파라미터가 없어, 뉴스가 아주 많은
회사에서는 '최근 1년 전체'가 아니라 '최신순 상한(~1000) + 정확도순 보강'이 된다. 즉
고volume 회사는 1년 창의 '오래된 꼬리'(예: 8~12개월 전)가 최신순 경로로는 안 잡힐 수
있으며, 이는 숨김이 아니라 표기되는 '수집 완전성의 한계'다(판단·깔때기 안전성과는 무관).

넓게 모으면 같은 사건의 중복 도배와 명백한 무관이 섞인다. 이를 판단(7축) '앞'에서
두 단계로 정리하되, 정리 = 삭제가 아니라 접힘이다(원칙 7):
- 중복 정리: '같은 하나의 사건'을 다룬 기사들을 한 대표로 묶는다. 대표만 판단으로
  보내고 나머지는 '중복 · 대표에 통합'으로 접는다(접근 가능). 같은 사건인지의 판단은
  더 싼 모델(Haiku)이 기사를 이해해서 한다 — 문자열 유사도 임계값·키워드가 아니다
  (원칙 1). 애매하면 합치지 않는다(다른 사건을 합쳐 하나를 없애지 않도록).
- 간이 관련성 관문: 재무·감사와 '명백히' 무관한 것만 접는다(광고·연예·봉사 등).
  느슨한 관문이라 애매하면 통과시킨다 — 진짜 악재를 앞단에서 떨구지 않기 위함(원칙 7).
- 양 조절은 계정 필터가 아니라 이 입구 정리(중복·명백무관)와 심각도 임계치로 한다.

## 접힘은 삭제가 아니다 (원칙 7 — 절대)
- 접힌 뉴스는 데이터에서 지우지 않는다. 맨 아래 "표시되지 않은 뉴스 N건 ▸" 한 줄로
  접되, 펼치면 전부(각각 '왜 접혔는지' 라벨과 함께) 보인다.
- 접힘 사유는 판단 후(단순 호재·무관·중립·기권)와 판단 전 입구(중복·간이무관) 둘 다이며,
  둘 다 하단에 보존된다. 정합: 수집 N = 표시 + 접힘(판단후 + 입구). 아무것도 증발하지 않는다.
- 수집·표시·접힘 건수를 화면에 함께 표기한다("수집 N · 표시 M · 접힘 K"). 감사인이
  무엇이 걸러졌는지 인지·검증할 수 있어야 한다(블랙박스 금지 — 투명성).
- 재무·감사 '악재'가 계정 미배정이면 여전히 기본 화면 ② 공용 큐에 보인다(위험
  미배정은 숨기지 않는다). 접히는 것은 위험 신호가 아닌 것(단순 호재·무관·기권)뿐이다.
- 남는 공백(관측): 비법적 위험 해소(예: 워크아웃 졸업)는 stage 축에 안 담겨 '단순
  호재'로 접힐 수 있다. 이는 표시층이 아니라 판단층(새 판단 필드)의 문제다.
