# 출시 기준 (release-criteria.md)

> 이 문서는 "무엇이 고쳐지면 출시할 수 있는가", 곧 종료 조건을 정의한다.
> 헌법(principles.md)이 '무엇이 옳은가'라면, 이 문서는 그 위에서 '무엇이 출시를
> 막고 무엇이 막지 않는가'를 가른다. 원칙에 종속되며, 원칙과 충돌하면 이 문서가
> 잘못된 것이다.
>
> 배경: 루프를 거듭하며 "발견 → 수정 → 새 발견"이 반복돼 종료 조건이 흐려졌다.
> 감사인이 "이렇게 고쳐서 끝이 있나"를 물었고, 아래 선으로 답한다. 완벽한 판단은
> 애초에 목표가 아니었다 — 목표였다면 이 도구의 정체성에 "판정은 감사인이 한다"를
> 박지 않았을 것이다.

## 대원칙: 출시를 가르는 선 = "감사인이 화면에서 알아챌 수 있는가"

이 도구의 정체성은 "위험을 판정하지 않는다 — 판정은 감사인이 한다"이다. 그러므로
출시 가능성은 '판단이 완벽한가'가 아니라 '감사인이 판정할 기회를 온전히 보장받는가'로
가른다. 판정의 재료(뉴스·계정·근거·분모)를 오염·소실시켜 판정 기회 자체를 박탈하는
것만이 출시를 막는다. 판정 자체의 미세한 흔들림은, 근거가 표시되어 감사인이 스스로
바로잡을 수 있는 한, 출시를 막지 않는다.

## 1. 출시를 막는 것 = 구조적 결함 (blocker)

감사인이 화면의 근거를 아무리 읽어도 알아챌 수 없는 결함. 판정 기회 자체가 박탈되는
것. 발견 즉시 고친다.

- **뉴스 증발** — `수집 = 표시 + 접힘` 항등식이 깨져 뉴스가 화면 어디에도 없이 사라진다.
- **계정 증발** — 원본 재무제표 행 수 ≠ 화면 계정 수. 회사가 보고한 계정이 조용히 빠진다.
- **서로 다른 사건의 오병합** — 중복 제거·사건군 묶기가 별개 사안을 하나로 삼켜 하나를 없앤다.
- **하드코딩·고정 목록·점수 공식** — 사건을 단어/유형표로 분류하거나 축에 가중치를 곱한다(원칙 1).
- **계정 출처가 회사 보고 명칭이 아닌 것** — 화면 계정명이 우리가 지어낸 라벨이다(울타리 위반).
- **재무 분모가 최신 보고서가 아닌 것** — 규모 비율의 분모가 낡은 보고서라 '가짜 정밀'을 만든다.
- **키·캐시 노출** — 비밀(API 키 등)이 파일·로그·화면·저장소에 샌다.

▷ 근거: 위 일곱은 공통적으로 '감사인이 화면을 봐도 모른다'. 뉴스가 사라졌는지, 계정이
빠졌는지, 두 사건이 합쳐졌는지, 분모가 작년 것인지는 화면에 근거로 드러나지 않는다.
판정의 재료 자체가 오염·소실되므로 감사인의 검증(원칙 5)이 작동할 수 없다. 이는
원칙 7(숨기지 않음)·원칙 1(하드코딩 금지)·원칙 4(가짜 정밀 금지)의 물리적 경계다.

▷ 판별: "감사인이 이 결함의 존재를 화면 근거만으로 알아챌 수 있는가?" 없으면 blocker다.

## 2. 출시를 막지 않는 것 = 판단 품질의 미세 편차 (관측)

감사인이 화면에 표시된 근거를 읽고 스스로 판정할 수 있는 것. 관측 대상이지 blocker가
아니다. 값싸게 고칠 수 있으면 고치되, 없다고 출시를 막지 않는다.

- **심각도 순서의 미세한 어색함** — 어느 악재가 몇 번째냐의 흔들림.
- **라벨이 다소 많거나 적음** — 예: '감사 확인 요망'이 변별력을 조금 잃음.
- **방향(악재/호재/중립) 판정의 경계 사례 흔들림** — 혼재 신호(호재+악재)의 종합.
- **판단 이유와 노출 위치의 불일치** — 예: 이유에 "소규모"라 쓰고 상단에 노출.

▷ 근거: 이 도구는 결론이 아니라 '가설 + 근거'를 낸다(원칙 3). 화면에는 제목·심각도
위치·계정 태그·한 줄 이유(1층)와 축별 근거·원문 문장(2층)이 함께 나온다(원칙 5).
그래서 순서가 한 칸 어긋나거나 라벨이 하나 더 붙어도 감사인은 근거를 읽고 수초 내
스스로 판정한다 — 판정 기회가 살아 있다. 완벽한 판단은 목표가 아니다. 목표였다면
"판정은 감사인이 한다"를 정체성에 박지 않았을 것이다. 원칙 5가 이미 이 영역을 덮는다.

▷ 주의: '막지 않는다'가 '무시한다'는 아니다. 이런 편차는 harness/pending/에 관측으로
남겨 판단력 개선(프롬프트·검증방 판단)으로 다룬다. 다만 그 개선이 완료되기 전에도
출시할 수 있다는 뜻이다.

## 3. 판별 질문 — 새 이슈가 나왔을 때 적용

새 결함·관측이 나오면 하나만 묻는다:

  **"감사인이 화면을 보고 이 문제를 알아챌 수 있는가?"**

- 알아챌 수 **없다** → 출시 blocker. 즉시 고친다(§1).
- 알아챌 수 **있다** → 관측 대상. harness/pending/에 기록하되 출시를 막지 않는다(§2).

이 질문은 '판단이 틀렸는가'를 묻지 않는다 — '감사인이 스스로 바로잡을 수 있는가'를
묻는다. 틀린 판단이라도 근거가 표시돼 감사인이 바로잡을 수 있으면 §2다. 옳은
동작이라도 감사인이 그 존재를 화면에서 알 수 없으면(예: 소리 없는 증발) §1이다.

## 4. 일반성 검증 방법 — 특정 회사 맞춤이 아님을 값싸게 증명 (원칙 2)

특정 회사에서 발견된 결함을 고쳤을 때, 그 수정이 '그 회사 맞춤'(원칙 2 위반)이 아님을
증명해야 한다. 회사를 여럿 라이브로 돌리는 대신, 더 값싸고 더 강한 방법을 쓴다:

  **"다른 회사 데이터에 같은 병리를 주입해, 같은 수정이 그 데이터도 낫게 하는가."**

▷ 선례(루프11 검증방): CJ제일제당의 '매각예정 처분집단' 병리를 삼성전자 데이터에
주입하고, IFRS 앵커를 켠 것과 끈 것을 대조했다. 앵커가 켜지면 삼성 주입본도 똑같이
섹션이 닫혔다 → 수정이 'CJ 바이트'가 아니라 '메커니즘'임이 반례로 증명됐다. 라이브
비용 0으로 "CJ 맞춤이 아니다"를 입증한 것이다. 이 방법이 여러 회사 라이브보다 싸고 강하다.

## 5. 라이브 실행의 위치 — 출시 판정의 필수 조건이 아니다

라이브(실제 뉴스 수집 + LLM 판단)는 '판단 품질(§2)' 확인용이며, 출시 판정의 필수
조건이 아니다. 구조적 결함(§1)은 캐시된 원본과 결정론적 재현으로 검증한다 — 증발·
오병합·계정 보존·하드코딩·비밀 노출은 API 키 없이 실측할 수 있다. 라이브는 실비용이
발생하므로, 라이브로만 확인 가능한 항목(§2의 판단 정확도)을 모아 한 번에 수행한다.

▷ 근거: §1이 전부 결정론으로 방어되면 출시할 수 있다. 라이브가 확인하는 것은 §2뿐이고
§2는 출시를 막지 않는다. 그래서 '크레딧이 없어 라이브를 못 했다'는 출시 불가 사유가
아니다. 단, 라이브 미검 항목은 정직히 남긴다(억지 통과 주장 금지, 원칙 4). 라이브
체크리스트를 모아 두는 예: `harness/pending/md-updates/loop13-live-verification-checklist.md`.
